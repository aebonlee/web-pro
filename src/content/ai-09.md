# 9장. FastAPI 백엔드 완성

이 장에서는 AI 모델 서빙을 넘어 **FastAPI 자체를 백엔드 프레임워크로서 정공법으로** 다룹니다. 앞선 1~8장이 Hugging Face 모델을 API로 감싸는 데 초점을 맞췄다면, 이 장은 라우터 설계·데이터 검증·데이터베이스 연동·CRUD API·프론트엔드 연동·테스트·배포까지 **실무에서 쓰는 백엔드 한 사이클 전체**를 직접 만들어 봅니다.

최종 목표는 명확합니다. **User와 Item 두 리소스에 대한 완전한 CRUD API를 만들고, React 프론트엔드와 연동한 뒤, pytest로 검증하고 Docker로 배포**하는 것입니다. 코드를 그대로 따라 입력하면 Swagger UI에서 바로 테스트되는 완성형 백엔드를 손에 넣게 됩니다.

### 이 장의 학습 목표

- FastAPI의 라우터·HTTP 메서드·경로/쿼리 매개변수를 체계적으로 이해하고 `APIRouter`로 라우터를 분리합니다.
- Pydantic으로 요청/응답 스키마를 정의하고 입력 검증을 자동화합니다.
- SQLAlchemy ORM과 SQLite/MySQL을 연동하고 의존성 주입(`Depends`)으로 DB 세션을 관리합니다.
- Repository 패턴으로 CRUD 로직을 분리해 완전한 REST API를 완성합니다.
- CORS를 설정하고 React와 연동해 목록·등록·수정·삭제 UI를 구현합니다.
- pytest + TestClient로 API를 테스트하고 Dockerfile로 컨테이너 배포합니다.

> **대상 독자** Python 기본 문법을 익힌 학습자. 데이터베이스는 개발 단계에서 SQLite(설치 불필요), 운영 단계에서 MySQL을 가정합니다.

## 9.1 FastAPI 개요와 개발 환경 설정

### 9.1.1 FastAPI란?

FastAPI는 Python 3.7+ 기반의 현대적인 웹 프레임워크로, **빠른 성능**과 **자동 문서화**가 가장 큰 특징입니다. 타입 힌트(Type Hint)를 그대로 활용해 입력을 검증하고, 코드를 작성하기만 하면 Swagger UI와 ReDoc 문서가 자동으로 생성됩니다.

Python 진영의 대표 프레임워크 셋을 비교하면 FastAPI의 위치가 분명해집니다.

| 구분 | Flask | Django | FastAPI |
| --- | --- | --- | --- |
| 타입 | 마이크로 프레임워크 | 풀스택 프레임워크 | API 특화 프레임워크 |
| 속도 | 중간 | 중간 | 매우 빠름 (Starlette 기반) |
| 자동 문서화 | 없음 (플러그인 필요) | 없음 (DRF 별도) | Swagger / ReDoc 내장 |
| 비동기 지원 | 제한적 | Django 3.1+ | 완전 지원 (async/await) |
| 데이터 검증 | 없음 | Form + Serializer | Pydantic 내장 |
| 학습 곡선 | 쉬움 | 높음 | 중간 (직관적) |

핵심은 **"타입 선언 = 검증 + 문서"** 라는 점입니다. 함수 인자에 `user_id: int`라고 쓰는 것만으로 타입 검증과 문서화가 동시에 이루어집니다.

### 9.1.3 첫 실행과 자동 문서 확인

가장 단순한 `main.py`로 서버가 뜨는지 확인합니다.

```python
# main.py
from fastapi import FastAPI

app = FastAPI()

@app.get("/")
def root():
    return {"message": "FastAPI 백엔드 정상 동작 중"}
```

```bash
# 서버 실행 (--reload: 코드 변경 시 자동 재시작)
uvicorn main:app --reload
```

`uvicorn main:app`은 "`main.py` 파일의 `app` 객체를 실행하라"는 의미입니다. 서버가 뜨면 브라우저에서 다음 주소로 확인합니다.

- **Swagger UI** : `http://127.0.0.1:8000/docs` — 엔드포인트를 직접 클릭해 테스트
- **ReDoc** : `http://127.0.0.1:8000/redoc` — 읽기 좋은 문서 형식
- **JSON 직접** : `http://127.0.0.1:8000/` — 응답 확인

코드를 한 줄도 더 쓰지 않았는데 `/docs`에 인터랙티브 문서가 생긴다는 점이 FastAPI의 첫 번째 감동 포인트입니다.

## 9.2 라우터와 HTTP 메서드

### 9.2.1 HTTP 메서드와 데코레이터

REST API는 같은 리소스에 대해 **HTTP 메서드**로 동작을 구분합니다. FastAPI는 메서드마다 전용 데코레이터를 제공합니다.

| 데코레이터 | HTTP 메서드 | 용도 | 예시 URL |
| --- | --- | --- | --- |
| `@app.get()` | GET | 데이터 조회 | `GET /users`, `GET /users/{id}` |
| `@app.post()` | POST | 데이터 생성 | `POST /users` |
| `@app.put()` | PUT | 전체 수정 | `PUT /users/{id}` |
| `@app.patch()` | PATCH | 부분 수정 | `PATCH /users/{id}` |
| `@app.delete()` | DELETE | 데이터 삭제 | `DELETE /users/{id}` |

### 9.2.2 경로 매개변수와 쿼리 매개변수

URL로 값을 전달하는 두 가지 방식을 구분하는 것이 라우팅의 기본입니다.

- **Path Parameter** : URL 경로에 값이 포함되는 방식 (`/users/42`)
- **Query Parameter** : URL 뒤에 `?key=value`로 전달 (`/items?skip=0&limit=5`)

```python
# main.py
from fastapi import FastAPI
from typing import Optional

app = FastAPI()

# [Path Parameter] URL 경로에 값이 포함되는 방식
# 예: GET /users/42 → user_id = 42
@app.get("/users/{user_id}")
def get_user(user_id: int):   # int 선언 시 타입 자동 검증
    return {"user_id": user_id}

# [Query Parameter] URL 뒤에 ?key=value 형태로 전달
# 예: GET /items?skip=0&limit=5&keyword=노트북
@app.get("/items")
def get_items(
    skip: int = 0,
    limit: int = 10,
    keyword: Optional[str] = None
):
    return {"skip": skip, "limit": limit, "keyword": keyword}

# [혼합] Path + Query 동시 사용
# 예: GET /users/1/items?active=false
@app.get("/users/{user_id}/items")
def get_user_items(user_id: int, active: bool = True):
    return {"user_id": user_id, "active": active}
```

규칙은 단순합니다. **함수 인자가 경로의 `{...}`에 있으면 Path 매개변수, 없으면 Query 매개변수**가 됩니다. 기본값을 주면 선택적 쿼리 매개변수가 됩니다.

### 9.2.3 APIRouter로 라우터 분리

엔드포인트가 늘어나면 `main.py` 한 파일이 비대해집니다. `APIRouter`로 리소스별 라우터를 분리하면 유지보수가 쉬워집니다.

```python
# routers/user.py
from fastapi import APIRouter

router = APIRouter(prefix="/users", tags=["users"])

@router.get("/")
def list_users():
    return [{"id": 1, "name": "Alice"}, {"id": 2, "name": "Bob"}]

@router.get("/{user_id}")
def get_user(user_id: int):
    return {"id": user_id, "name": "Alice"}
```

`prefix="/users"`를 주면 이 라우터의 모든 경로 앞에 `/users`가 자동으로 붙습니다. `tags=["users"]`는 Swagger 문서에서 엔드포인트를 그룹으로 묶어 줍니다. 이렇게 만든 라우터는 뒤에서 `main.py`가 `app.include_router(...)`로 등록합니다.

```bash
uvicorn main:app --reload

# 브라우저 주소창 테스트
# 전체 목록 : http://127.0.0.1:8000/users/
# 단건 조회 : http://127.0.0.1:8000/users/1
# 쿼리 파라미터: http://127.0.0.1:8000/items?skip=0&limit=5&keyword=노트북
# Swagger UI에서 POST/PUT/DELETE 테스트:
#   http://127.0.0.1:8000/docs → 엔드포인트 클릭 → Try it out
```

## 9.3 Pydantic 데이터 검증

### 9.3.1 schemas.py — 요청/응답 스키마 정의

Pydantic 모델은 **요청 본문의 형식을 강제하고, 응답 형식을 정의**하는 "데이터 계약서"입니다. `schemas.py`는 단독 실행되지 않고 `main.py`나 `routers/*.py`에서 import해 사용하는 부품 파일입니다.

```python
# schemas.py ← 프로젝트 루트에 생성
from pydantic import BaseModel, EmailStr, Field, field_validator
from typing import Optional
from datetime import datetime

# ── User 스키마 ──────────────────────────────
class UserCreate(BaseModel):
    username: str = Field(..., min_length=3, max_length=20)
    email: EmailStr
    password: str = Field(..., min_length=8)

    @field_validator("username")
    @classmethod
    def username_alphanumeric(cls, v):
        if not v.isalnum():
            raise ValueError("영문자/숫자만 가능합니다")
        return v

class UserResponse(BaseModel):
    id: int
    username: str
    email: str
    created_at: datetime
    model_config = {"from_attributes": True}  # ORM 객체 자동 변환

# ── Item 스키마 ──────────────────────────────
class ItemCreate(BaseModel):
    name: str = Field(..., min_length=1, max_length=100)
    price: float = Field(..., gt=0)

class ItemUpdate(BaseModel):
    name: Optional[str] = None    # None이면 수정 안 함
    price: Optional[float] = None

class ItemResponse(BaseModel):
    id: int
    name: str
    price: float
    owner_id: Optional[int] = None
    model_config = {"from_attributes": True}
```

각 요소의 의미를 짚어 봅니다.

- `Field(..., min_length=3)` : `...`은 "필수 값"이라는 뜻이고, `min_length`·`max_length`·`gt`(0보다 큼) 등으로 제약을 건다.
- `EmailStr` : 이메일 형식을 자동 검증하는 특수 타입 (`pydantic[email]` 설치 필요).
- `@field_validator` : 내장 규칙으로 부족할 때 직접 검증 함수를 추가한다.
- `model_config = {"from_attributes": True}` : SQLAlchemy ORM 객체를 그대로 응답 모델로 변환할 수 있게 한다 (이전 버전의 `orm_mode`).

> **Create와 Response를 나누는 이유** `UserCreate`에는 `password`가 있지만 `UserResponse`에는 없습니다. **입력으로는 비밀번호를 받되, 응답에는 절대 노출하지 않기** 위해서입니다. 또한 응답에만 `id`·`created_at`처럼 서버가 생성하는 값을 둡니다.

### 9.3.2 검증 동작 확인

`http://127.0.0.1:8000/docs → POST /users → Try it out`에서 직접 확인할 수 있습니다.

```text
# [정상 입력]
{ "username": "alice123", "email": "alice@example.com", "password": "mypassword" }

# [검증 실패 → 422 자동 반환]
# username: "a"      → min_length=3 위반
# username: "ali!!"  → 영문자/숫자 외 문자 포함
# email: "notmail"   → 이메일 형식 위반
# password: "1234"   → min_length=8 위반
```

검증에 실패하면 FastAPI가 **자동으로 422 Unprocessable Entity** 응답을 보내고, 어떤 필드가 왜 실패했는지 친절한 메시지까지 만들어 줍니다. 개발자가 `if` 검증 코드를 한 줄도 쓰지 않아도 됩니다.

## 9.4 SQLAlchemy와 데이터베이스 연동

### 9.4.1 DB 선택 가이드

| 항목 | SQLite (개발용) | MySQL (운영용) |
| --- | --- | --- |
| 용도 | 개발/테스트/소규모 | 운영 환경/대용량 |
| 설치 | 별도 설치 불필요 | MySQL 서버 필요 |
| Connection URL | `sqlite:///./app.db` | `mysql+pymysql://user:pw@host/db` |
| 파일 방식 | 단일 `.db` 파일 | 서버-클라이언트 방식 |
| 실습 추천 | ✅ 로컬 개발 | ✅ 운영 배포 |

개발 단계에서는 설치가 필요 없는 SQLite로 시작하고, 운영 단계에서 접속 URL만 MySQL로 바꾸면 됩니다. 코드의 나머지는 그대로 둬도 됩니다 — 이것이 ORM을 쓰는 큰 이점입니다.

### 9.4.2 database.py — DB 연결 설정

```python
# database.py ← 프로젝트 루트에 생성
from sqlalchemy import create_engine
from sqlalchemy.orm import declarative_base, sessionmaker

# SQLite (개발용) — 별도 설치 없이 바로 사용 가능
DATABASE_URL = "sqlite:///./app.db"

# MySQL (운영용) — 아래 주석 해제 후 DATABASE_URL 교체
# import os
# from dotenv import load_dotenv
# load_dotenv()
# DATABASE_URL = (
#     f"mysql+pymysql://{os.getenv('DB_USER')}:{os.getenv('DB_PASS')}"
#     f"@{os.getenv('DB_HOST')}/{os.getenv('DB_NAME')}"
# )

engine = create_engine(
    DATABASE_URL,
    connect_args={"check_same_thread": False}  # SQLite 전용 옵션
)
SessionLocal = sessionmaker(autocommit=False, autoflush=False, bind=engine)
Base = declarative_base()

# DB 세션 의존성 주입용 제너레이터 — 라우터에서 Depends(get_db)로 사용
def get_db():
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()
```

`get_db`는 이 구조의 핵심입니다. 요청이 들어오면 세션을 열고(`yield db`), 요청 처리가 끝나면 `finally`에서 반드시 닫습니다(`db.close()`). 라우터 함수에서 `db: Session = Depends(get_db)`라고 쓰면 FastAPI가 이 제너레이터를 자동으로 호출해 세션을 주입하고, 응답이 끝나면 정리까지 해 줍니다. **세션 누수를 구조적으로 막는 패턴**입니다.

### 9.4.3 models.py — ORM 모델 정의

```python
# models.py ← 프로젝트 루트에 생성
from sqlalchemy import Column, Integer, String, Float, Boolean, DateTime, ForeignKey
from sqlalchemy.orm import relationship
from sqlalchemy.sql import func
from database import Base

class User(Base):
    __tablename__ = "users"

    id         = Column(Integer, primary_key=True, index=True)
    username   = Column(String(50), unique=True, nullable=False)
    email      = Column(String(100), unique=True, nullable=False)
    password   = Column(String(200), nullable=False)
    is_active  = Column(Boolean, default=True)
    created_at = Column(DateTime(timezone=True), server_default=func.now())

    items = relationship("Item", back_populates="owner")

class Item(Base):
    __tablename__ = "items"

    id       = Column(Integer, primary_key=True, index=True)
    name     = Column(String(100), nullable=False)
    price    = Column(Float, nullable=False)
    owner_id = Column(Integer, ForeignKey("users.id"), nullable=True)

    owner = relationship("User", back_populates="items")
```

`relationship`과 `ForeignKey`로 **1:N 관계**(한 User가 여러 Item을 가짐)를 표현합니다. `back_populates`는 양쪽 모델이 서로를 참조하도록 연결해, `user.items`로 사용자의 아이템 목록을, `item.owner`로 아이템의 주인을 바로 얻을 수 있게 합니다.

```bash
# main.py에 아래 코드 추가 후 서버 실행하면 테이블 자동 생성
uvicorn main:app --reload

# 프로젝트 루트에 app.db 파일이 생성됩니다
# DB 내용 확인: DB Browser for SQLite (무료 GUI)
# 다운로드: https://sqlitebrowser.org/
```

> **스키마와 모델은 다르다** `schemas.py`(Pydantic)는 "API가 주고받는 데이터의 형식"이고, `models.py`(SQLAlchemy)는 "DB 테이블의 구조"입니다. 둘을 분리해 두면 API 표현과 저장 구조를 독립적으로 바꿀 수 있습니다. (예: DB에 `password`는 저장하되 응답 스키마엔 빼는 것)

## 9.5 CRUD API 완성 — Repository 패턴

이 절의 목표는 **User와 Item 두 리소스에 대한 완전한 CRUD API**를 구현하는 것입니다. DB 처리 로직(`crud/`)과 라우팅 로직(`routers/`)을 분리하는 **Repository 패턴**으로 만듭니다. 라우터는 "요청을 받고 응답을 돌려주는 일"에만 집중하고, 실제 DB 조작은 crud 함수가 담당합니다.

### 9.5.1 crud/user_crud.py — User CRUD 함수

```python
# crud/user_crud.py
from sqlalchemy.orm import Session
from models import User
from schemas import UserCreate
from passlib.context import CryptContext

pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")

def get_user(db: Session, user_id: int):
    return db.query(User).filter(User.id == user_id).first()

def get_user_by_email(db: Session, email: str):
    return db.query(User).filter(User.email == email).first()

def get_users(db: Session, skip: int = 0, limit: int = 100):
    return db.query(User).offset(skip).limit(limit).all()

def create_user(db: Session, user: UserCreate):
    hashed_pw = pwd_context.hash(user.password)
    db_user = User(username=user.username, email=user.email, password=hashed_pw)
    db.add(db_user)
    db.commit()
    db.refresh(db_user)
    return db_user

def delete_user(db: Session, user_id: int):
    user = get_user(db, user_id)
    if user:
        db.delete(user)
        db.commit()
    return user
```

비밀번호는 **절대 평문으로 저장하지 않습니다.** `passlib`의 `CryptContext`로 bcrypt 해싱한 값을 저장합니다. `db.add → db.commit → db.refresh` 3단계는 "메모리에 추가 → DB에 반영 → 생성된 id 등 최신 값을 객체에 다시 로드"하는 표준 흐름입니다.

### 9.5.2 crud/item_crud.py — Item CRUD 함수

```python
# crud/item_crud.py
from sqlalchemy.orm import Session
from models import Item
from schemas import ItemCreate, ItemUpdate

def get_item(db: Session, item_id: int):
    return db.query(Item).filter(Item.id == item_id).first()

def get_items(db: Session, skip: int = 0, limit: int = 100):
    return db.query(Item).offset(skip).limit(limit).all()

def create_item(db: Session, item: ItemCreate, owner_id: int = None):
    db_item = Item(name=item.name, price=item.price, owner_id=owner_id)
    db.add(db_item)
    db.commit()
    db.refresh(db_item)
    return db_item

def update_item(db: Session, item_id: int, item_data: ItemUpdate):
    db_item = get_item(db, item_id)
    if not db_item:
        return None
    # None이 아닌 값만 업데이트 (부분 수정 허용)
    if item_data.name is not None:
        db_item.name = item_data.name
    if item_data.price is not None:
        db_item.price = item_data.price
    db.commit()
    db.refresh(db_item)
    return db_item

def delete_item(db: Session, item_id: int):
    db_item = get_item(db, item_id)
    if db_item:
        db.delete(db_item)
        db.commit()
    return db_item
```

`update_item`은 **부분 수정**을 우아하게 처리합니다. `ItemUpdate`의 필드가 `None`이면 그 값은 건드리지 않고, 전달된 값만 갱신합니다. 덕분에 이름만 바꾸거나 가격만 바꾸는 요청을 같은 함수로 처리할 수 있습니다.

### 9.5.3 routers/user.py — User 라우터

```python
# routers/user.py
from fastapi import APIRouter, Depends, HTTPException, status
from sqlalchemy.orm import Session
from database import get_db
import crud.user_crud as crud
import schemas

router = APIRouter(prefix="/users", tags=["users"])

@router.get("/", response_model=list[schemas.UserResponse])
def list_users(skip: int = 0, limit: int = 10, db: Session = Depends(get_db)):
    return crud.get_users(db, skip=skip, limit=limit)

@router.get("/{user_id}", response_model=schemas.UserResponse)
def get_user(user_id: int, db: Session = Depends(get_db)):
    user = crud.get_user(db, user_id)
    if not user:
        raise HTTPException(status_code=404, detail="사용자를 찾을 수 없습니다")
    return user

@router.post("/", response_model=schemas.UserResponse, status_code=201)
def create_user(user: schemas.UserCreate, db: Session = Depends(get_db)):
    existing = crud.get_user_by_email(db, user.email)
    if existing:
        raise HTTPException(status_code=400, detail="이미 등록된 이메일입니다")
    return crud.create_user(db, user)

@router.delete("/{user_id}", status_code=204)
def delete_user(user_id: int, db: Session = Depends(get_db)):
    if not crud.delete_user(db, user_id):
        raise HTTPException(status_code=404, detail="사용자를 찾을 수 없습니다")
```

`response_model`을 지정하면 **응답이 그 스키마 형식으로 자동 변환·검증**되고, Swagger 문서에도 응답 형식이 표시됩니다. `UserResponse`에는 `password`가 없으므로 비밀번호가 응답에 새어 나가는 일이 구조적으로 차단됩니다. 없는 리소스를 찾으면 `HTTPException`으로 적절한 상태 코드를 던집니다.

### 9.5.6 HTTP 상태 코드 정리

REST API는 결과를 **상태 코드**로 전달합니다. 자주 쓰는 코드를 정리합니다.

| 상태 코드 | 의미 | 사용 시점 |
| --- | --- | --- |
| 200 OK | 성공 | GET, PUT 성공 |
| 201 Created | 생성 성공 | POST로 리소스 생성 |
| 204 No Content | 성공 (내용 없음) | DELETE 성공 |
| 400 Bad Request | 잘못된 요청 | 비즈니스 로직 오류 (중복 등) |
| 404 Not Found | 리소스 없음 | 존재하지 않는 데이터 조회 |
| 422 Unprocessable | 검증 실패 | Pydantic 타입/값 오류 (자동) |
| 500 Internal Error | 서버 오류 | 예상 못 한 예외 |

```text
# Swagger UI에서 전체 CRUD 테스트
# http://127.0.0.1:8000/docs 접속 후 순서대로 테스트
# ① POST /items   → {"name": "노트북", "price": 1500000} → 201 확인
# ② GET  /items   → 목록 조회, 등록된 아이템 확인
# ③ PUT  /items/1 → {"name": "게이밍 노트북", "price": 1800000} → 수정 확인
# ④ GET  /items/1 → 수정된 데이터 확인
# ⑤ DELETE /items/1 → 204 확인 → GET /items로 삭제 확인
```

## 9.7 React CRUD UI 구현

이 절의 목표는 3개 파일(`ItemList.jsx`, `ItemForm.jsx`, `App.jsx`)을 만들어 **복사 후 바로 동작하는 완전한 CRUD UI**를 완성하는 것입니다.

### 9.7.1 src/components/ItemList.jsx — 목록 & 삭제

```javascript
// src/components/ItemList.jsx
import { useEffect, useState } from "react";
import { fetchItems, deleteItem } from "../api/itemApi";

function ItemList({ onEdit, refreshKey }) {
  const [items, setItems] = useState([]);
  const [error, setError] = useState(null);   // 에러 상태 추가
  const [loading, setLoading] = useState(false);

  // refreshKey가 바뀔 때마다 목록 새로고침
  useEffect(() => {
    loadItems();
  }, [refreshKey]);

  const loadItems = async () => {
    setLoading(true);
    setError(null);
    try {
      const data = await fetchItems();
      setItems(data);
    } catch (e) {
      setError("서버에 연결할 수 없습니다. FastAPI 서버가 실행 중인지 확인하세요.");
    } finally {
      setLoading(false);
    }
  };

  const handleDelete = async (id) => {
    if (!window.confirm("삭제하시겠습니까?")) return;
    try {
      await deleteItem(id);
      loadItems();  // 삭제 후 목록 새로고침
    } catch (e) {
      alert("삭제에 실패했습니다.");
    }
  };

  if (loading) return <p>불러오는 중...</p>;
  if (error)   return <p style={{ color: "red" }}>{error}</p>;

  return (
    <div>
      <h2>아이템 목록</h2>
      {items.length === 0 && <p>등록된 아이템이 없습니다.</p>}
      <ul style={{ listStyle: "none", padding: 0 }}>
        {items.map(item => (
          <li key={item.id} style={{ padding: "8px", borderBottom: "1px solid #eee" }}>
            <strong>{item.name}</strong> — {item.price.toLocaleString()}원
            <button onClick={() => onEdit(item)}
              style={{ marginLeft: 12, padding: "4px 10px" }}>수정</button>
            <button onClick={() => handleDelete(item.id)}
              style={{ marginLeft: 4, padding: "4px 10px", color: "red" }}>삭제</button>
          </li>
        ))}
      </ul>
    </div>
  );
}

export default ItemList;
```

핵심은 **로딩·에러·빈 목록 상태를 모두 화면에 반영**한다는 점입니다. `refreshKey`가 바뀔 때마다 `useEffect`가 다시 실행되어 목록을 새로고침합니다 — 등록/수정/삭제 후 부모가 이 키를 바꿔 목록을 갱신합니다.

### 9.7.2 src/components/ItemForm.jsx — 등록 & 수정 폼

```javascript
// src/components/ItemForm.jsx
import { useState, useEffect } from "react";
import { createItem, updateItem } from "../api/itemApi";

// editItem prop이 있으면 수정 모드, 없으면 등록 모드
function ItemForm({ editItem, onSuccess }) {
  const [name, setName]   = useState("");
  const [price, setPrice] = useState("");
  const [submitting, setSubmitting] = useState(false);

  // editItem이 바뀔 때마다 폼 초기값 설정
  useEffect(() => {
    if (editItem) {
      setName(editItem.name);
      setPrice(String(editItem.price));
    } else {
      setName("");
      setPrice("");
    }
  }, [editItem]);

  const handleSubmit = async (e) => {
    e.preventDefault();
    if (!name.trim() || !price) return;
    setSubmitting(true);
    try {
      const payload = { name: name.trim(), price: Number(price) };
      if (editItem) {
        await updateItem(editItem.id, payload);  // PUT /items/{id}
      } else {
        await createItem(payload);               // POST /items
      }
      onSuccess();  // 부모에게 완료 알림 → 목록 새로고침
    } catch (e) {
      alert("저장에 실패했습니다. 입력값을 확인하세요.");
    } finally {
      setSubmitting(false);
    }
  };

  return (
    <div style={{ marginBottom: 24, padding: 16, background: "#f9f9f9", borderRadius: 8 }}>
      <h2>{editItem ? "아이템 수정" : "아이템 등록"}</h2>
      <form onSubmit={handleSubmit}>
        <input value={name} onChange={e => setName(e.target.value)}
          placeholder="이름" required
          style={{ marginRight: 8, padding: "6px 10px" }} />
        <input value={price} onChange={e => setPrice(e.target.value)}
          placeholder="가격" type="number" min="1" required
          style={{ marginRight: 8, padding: "6px 10px" }} />
        <button type="submit" disabled={submitting}
          style={{ padding: "6px 16px", background: "#2E75B6", color: "#fff", border: "none", borderRadius: 4 }}>
          {submitting ? "저장 중..." : editItem ? "수정 완료" : "등록"}
        </button>
        {editItem && (
          <button type="button" onClick={onSuccess}
            style={{ marginLeft: 8, padding: "6px 12px" }}>취소</button>
        )}
      </form>
    </div>
  );
}

export default ItemForm;
```

하나의 폼이 **등록과 수정을 모두 담당**합니다. `editItem` prop의 유무로 모드를 구분하고, `editItem`이 바뀌면 폼에 기존 값을 채워 넣습니다.

### 9.7.3 src/App.jsx — 최종 조합

```javascript
// src/App.jsx
import { useState } from "react";
import ItemList from "./components/ItemList";
import ItemForm from "./components/ItemForm";

function App() {
  const [editItem, setEditItem] = useState(null);
  const [refreshKey, setRefreshKey] = useState(0);

  const handleSuccess = () => {
    setEditItem(null);            // 폼 초기화 (등록 모드로 복귀)
    setRefreshKey(k => k + 1);    // 목록 새로고침 트리거
  };

  return (
    <div style={{ maxWidth: 700, margin: "40px auto", padding: "0 20px", fontFamily: "sans-serif" }}>
      <h1 style={{ color: "#1E3A5F" }}>FastAPI + React 아이템 관리</h1>
      <ItemForm editItem={editItem} onSuccess={handleSuccess} />
      <ItemList refreshKey={refreshKey} onEdit={setEditItem} />
    </div>
  );
}

export default App;
```

`App`이 **상태를 들고 두 컴포넌트를 연결**합니다. `editItem`은 어떤 아이템을 수정 중인지, `refreshKey`는 목록 갱신 신호입니다. 등록·수정이 끝나면 `handleSuccess`가 폼을 초기화하고 목록을 새로고침합니다.

## 9.8 테스트와 Docker 배포

### 9.8.1 pytest + TestClient

FastAPI는 `TestClient`로 서버를 실제로 띄우지 않고도 엔드포인트를 테스트할 수 있습니다.

```python
# test_main.py ← 프로젝트 루트에 생성
from fastapi.testclient import TestClient
from main import app

client = TestClient(app)

def test_create_item():
    response = client.post("/items", json={"name": "테스트 상품", "price": 10000})
    assert response.status_code == 201
    data = response.json()
    assert data["name"] == "테스트 상품"
    assert data["price"] == 10000

def test_get_items():
    response = client.get("/items")
    assert response.status_code == 200
    assert isinstance(response.json(), list)

def test_update_item():
    # 먼저 생성
    create_res = client.post("/items", json={"name": "원래 이름", "price": 5000})
    item_id = create_res.json()["id"]
    # 수정
    update_res = client.put(f"/items/{item_id}", json={"name": "수정된 이름", "price": 9000})
    assert update_res.status_code == 200
    assert update_res.json()["name"] == "수정된 이름"

def test_delete_item():
    create_res = client.post("/items", json={"name": "삭제할 상품", "price": 1000})
    item_id = create_res.json()["id"]
    del_res = client.delete(f"/items/{item_id}")
    assert del_res.status_code == 204

def test_item_not_found():
    response = client.get("/items/99999")
    assert response.status_code == 404

def test_create_user():
    response = client.post("/users", json={
        "username": "testuser",
        "email": "test@example.com",
        "password": "securepass123"
    })
    assert response.status_code == 201
    assert "password" not in response.json()  # 비밀번호 노출 X

def test_get_user_not_found():
    response = client.get("/users/99999")
    assert response.status_code == 404
```

마지막 `assert "password" not in response.json()`가 중요합니다. **응답에 비밀번호가 새어 나가지 않는지**를 테스트로 보장합니다 — `UserResponse` 스키마 설계가 실제로 지켜지는지 자동 검증하는 것입니다.

```bash
pytest -v                      # 전체 테스트 실행 + 상세 출력
pytest test_main.py -v         # 특정 파일만 실행
pytest -v -k "test_create"     # 이름에 test_create 포함된 테스트만 실행
```

### 9.8.2 Dockerfile

애플리케이션과 실행 환경을 컨테이너로 패키징하면 어디서나 동일하게 실행됩니다.

```dockerfile
# Dockerfile ← 프로젝트 루트에 생성
FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
EXPOSE 8000
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
```

`requirements.txt`를 먼저 복사해 설치한 뒤 소스를 복사하는 순서가 핵심입니다. 소스만 바뀌면 의존성 설치 레이어가 캐시되어 빌드가 빨라집니다. `--host 0.0.0.0`은 컨테이너 외부에서 접속을 받기 위한 설정입니다.

```bash
docker build -t fastapi-app .          # 이미지 빌드
docker run -p 8000:8000 fastapi-app    # 컨테이너 실행

# 환경변수 파일(.env)과 함께 실행
docker run -p 8000:8000 --env-file .env fastapi-app

# 실행 중인 컨테이너 확인 및 중지
docker ps                              # 실행 중인 컨테이너 목록
docker stop <CONTAINER_ID>             # 컨테이너 중지
```

## 9.9 종합 실습 프로젝트 — 도서 관리 API

지금까지 배운 모든 것을 합쳐 **도서 관리 API**를 직접 설계·구현합니다. User 인증과 Book CRUD를 결합한 실전 프로젝트입니다.

### 9.9.1 프로젝트 명세

| 엔드포인트 | 메서드 | 설명 | 인증 필요 |
| --- | --- | --- | --- |
| `/auth/register` | POST | 회원가입 | ❌ |
| `/auth/login` | POST | 로그인 (JWT 발급) | ❌ |
| `/books` | GET | 도서 목록 조회 (페이지네이션) | ❌ |
| `/books/{id}` | GET | 도서 상세 조회 | ❌ |
| `/books` | POST | 도서 등록 | ✅ |
| `/books/{id}` | PUT | 도서 정보 수정 | ✅ (본인만) |
| `/books/{id}` | DELETE | 도서 삭제 | ✅ (본인만) |
| `/books/search` | GET | 제목/저자 검색 | ❌ |

### 9.9.2 DB 스키마

| 테이블 | 컬럼 | 타입 | 설명 |
| --- | --- | --- | --- |
| users | id | INT PK | 자동 증가 |
| users | username | VARCHAR(50) | 유니크 |
| users | email | VARCHAR(100) | 유니크 |
| users | password | VARCHAR(200) | bcrypt 해시 |
| books | id | INT PK | 자동 증가 |
| books | title | VARCHAR(200) | 도서 제목 |
| books | author | VARCHAR(100) | 저자 |
| books | price | FLOAT | 가격 |
| books | owner_id | INT FK | users.id 참조 |

### 9.9.3 구현 가이드

앞 절의 코드를 그대로 응용하면 됩니다. 매핑 관계는 다음과 같습니다.

- **models.py** : 9.4.3의 `Item` 모델을 `Book`(title·author·price·owner_id)으로 바꿔 정의합니다.
- **schemas.py** : 9.3의 `ItemCreate/ItemUpdate/ItemResponse` 패턴을 `BookCreate/BookUpdate/BookResponse`로 복제합니다.
- **crud/book_crud.py** : 9.5.2의 Item CRUD를 그대로 가져와 검색 함수(`db.query(Book).filter(Book.title.contains(keyword))`)를 추가합니다.
- **routers/book.py** : 9.5.4의 Item 라우터에 `/books/search`와 본인 확인 로직을 더합니다.
- **인증** : `/auth/login`에서 `python-jose`로 JWT를 발급하고, 수정·삭제 시 `owner_id`가 현재 사용자와 일치하는지 확인합니다.
