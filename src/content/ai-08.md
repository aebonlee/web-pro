## 8.1 FastAPI 운영 환경: Uvicorn과 Gunicorn

### (1) FastAPI 개발 vs 운영 환경 차이

FastAPI 개발 서버(`uvicorn --reload`)는 코드 변경을 자동 감지하고 재시작하는 편의 기능이 있지만, 단일 프로세스로만 동작하고 에러 처리가 느슨하여 실제 트래픽을 감당하기에 적합하지 않습니다. 운영 환경에서는 다중 워커(Multi-Worker) 프로세스, 구조화된 로깅, 안정적인 에러 처리, 보안 설정이 필수입니다.

| 구분 | 개발 환경 (Development) | 운영 환경 (Production) |
| --- | --- | --- |
| 실행 명령 | `uvicorn main:app --reload` | `gunicorn main:app -w 4 -k uvicorn.workers.UvicornWorker` |
| 워커 수 | 1개 (싱글 프로세스) | CPU 코어 × 2 + 1 (멀티 프로세스) |
| 자동 재시작 | 코드 변경 시 자동 재시작 | 크래시 발생 시에만 재시작 |
| 디버그 모드 | `debug=True`, 상세 오류 노출 | `debug=False`, 오류 은폐 |
| 로깅 | 콘솔 출력 (stdout) | 파일 로테이션 + 구조화 JSON 로그 |
| 보안 | CORS 전체 허용, HTTP | CORS 제한, HTTPS (Nginx), 인증 |
| 성능 | 낮음 (단일, 오버헤드 있음) | 높음 (멀티워커, 최적화) |

개발 환경과 운영 환경의 설정을 코드 레벨에서도 분리하는 것이 중요합니다. python-dotenv와 Pydantic Settings를 활용하면 환경변수 기반으로 설정을 손쉽게 관리할 수 있습니다.

```python
# config.py — Pydantic Settings로 환경별 설정 관리
from pydantic_settings import BaseSettings
from functools import lru_cache
from typing import Literal


class Settings(BaseSettings):
    # 앱 기본 설정
    APP_NAME: str = "AI API Service"
    APP_VERSION: str = "1.0.0"
    ENVIRONMENT: Literal["development", "staging", "production"] = "development"
    DEBUG: bool = False

    # 서버 설정
    HOST: str = "0.0.0.0"
    PORT: int = 8000
    WORKERS: int = 1

    # 모델 설정
    MODEL_ID: str = "runwayml/stable-diffusion-v1-5"
    MODEL_CACHE_DIR: str = "./model_cache"
    HF_TOKEN: str = ""

    # 보안 설정
    SECRET_KEY: str = "change-this-in-production"
    ALLOWED_ORIGINS: list = ["*"]
    API_KEY: str = ""

    # 로깅 설정
    LOG_LEVEL: str = "INFO"
    LOG_FILE: str = "./logs/app.log"

    class Config:
        env_file = ".env"
        env_file_encoding = "utf-8"

    @property
    def is_production(self) -> bool:
        return self.ENVIRONMENT == "production"

    @property
    def recommended_workers(self) -> int:
        import os
        return (os.cpu_count() or 1) * 2 + 1


@lru_cache()
def get_settings() -> Settings:
    return Settings()


# .env.development 예시
# ENVIRONMENT=development
# DEBUG=True
# LOG_LEVEL=DEBUG
# WORKERS=1


# .env.production 예시
# ENVIRONMENT=production
# DEBUG=False
# LOG_LEVEL=INFO
# WORKERS=5
# SECRET_KEY=supersecretkeyhere
# HF_TOKEN=hf_xxxxxxxxxxxx
# ALLOWED_ORIGINS=["https://myapp.com","https://www.myapp.com"]
```

### (2) Uvicorn 단독 실행의 한계와 Gunicorn 연동 이유

Uvicorn은 ASGI(Asynchronous Server Gateway Interface) 서버로, 단일 이벤트 루프에서 비동기 요청을 처리합니다. 비동기 I/O 작업에서는 매우 효율적이지만, CPU를 집중적으로 사용하는 AI 모델 추론에서는 싱글 프로세스가 병목이 됩니다. Gunicorn은 이 문제를 해결해주는 프리포크(pre-fork) 워커 매니저입니다.

#### Uvicorn 단독의 한계

1. **단일 프로세스 제약**: GIL(Global Interpreter Lock)으로 인해 CPU 바운드 작업이 다른 요청을 블로킹
2. **크래시 자동 복구 없음**: 워커 프로세스가 비정상 종료되면 서비스 전체 중단
3. **부드러운 재시작(Graceful Reload) 미지원**: 무중단 배포 불가
4. **CPU 코어 활용 불가**: 4코어 서버에서도 1코어만 사용

#### Gunicorn + UvicornWorker 조합의 장점

1. **멀티 프로세스**: 여러 Uvicorn 워커를 독립 프로세스로 실행 → 병렬 요청 처리
2. **자동 워커 재시작**: 워커 크래시 시 Gunicorn이 자동으로 새 워커 생성
3. **무중단 재시작**: `gunicorn --reload` 또는 SIGHUP 시그널로 순차 재시작
4. **워커 수 동적 조정**: 트래픽에 따라 `max_requests`, `timeout` 등 세밀한 조정 가능
5. **프로세스 격리**: 한 워커의 메모리 누수가 다른 워커에 영향 없음

```bash
# ── Uvicorn 단독 실행 (개발 전용) ───────────────────────
# 기본 개발 서버 — 단일 프로세스
uvicorn main:app --reload --host 0.0.0.0 --port 8000


# ── Gunicorn + UvicornWorker (운영 권장) ────────────────
# 기본 멀티워커 실행
gunicorn main:app \
  --worker-class uvicorn.workers.UvicornWorker \
  --workers 4 \
  --bind 0.0.0.0:8000


# ── Gunicorn 워커 수 자동 계산 ──────────────────────────
# 일반 권장식: CPU 코어 수 × 2 + 1
WORKERS=$(python3 -c "import os; print(os.cpu_count() * 2 + 1)")
echo "권장 워커 수: $WORKERS"


# AI 모델 서비스 특화: 메모리 집약적이므로 보수적으로 설정
# GPU 메모리가 제한적이면 워커 수를 줄여야 함
GPU_VRAM_GB=16
MODEL_VRAM_GB=4
MAX_WORKERS=$(($GPU_VRAM_GB / $MODEL_VRAM_GB))
echo "VRAM 기반 최대 워커 수: $MAX_WORKERS"


# ── 완전한 운영 실행 명령어 ─────────────────────────────
gunicorn main:app \
  --worker-class uvicorn.workers.UvicornWorker \
  --workers 4 \
  --bind 0.0.0.0:8000 \
  --timeout 120 \
  --keep-alive 5 \
  --max-requests 1000 \
  --max-requests-jitter 100 \
  --graceful-timeout 30 \
  --access-logfile ./logs/access.log \
  --error-logfile ./logs/error.log \
  --log-level info \
  --pid ./gunicorn.pid
```

### (3) 운영 환경 실행 명령어 및 설정 최적화

Gunicorn 설정을 명령행 인자로 반복 입력하는 대신, `gunicorn.conf.py` 설정 파일로 관리하면 훨씬 체계적이고 버전 관리가 편리합니다.

```python
# gunicorn.conf.py — Gunicorn 운영 설정 파일
import os
import multiprocessing


# ── 서버 소켓 ────────────────────────────────────────────
bind = "0.0.0.0:8000"
backlog = 2048    # 대기 중인 연결 수 (기본 2048)


# ── 워커 설정 ────────────────────────────────────────────
# AI 서비스: GPU 메모리 고려하여 워커 수 제한
# 일반 API: cpu_count * 2 + 1
cpu_count = multiprocessing.cpu_count()
workers = int(os.getenv("GUNICORN_WORKERS", min(cpu_count * 2 + 1, 4)))

worker_class = "uvicorn.workers.UvicornWorker"
worker_connections = 1000  # 워커당 최대 동시 연결 수 (async)

# 워커 수명 제한: 메모리 누수 방지 (1000요청 후 자동 재생성)
max_requests = 1000
max_requests_jitter = 100  # 동시 재시작 방지를 위한 랜덤 지연


# ── 타임아웃 설정 ────────────────────────────────────────
# AI 추론은 시간이 오래 걸릴 수 있으므로 넉넉히 설정
timeout = 120            # 워커 타임아웃 (초). AI 추론 시간 고려
graceful_timeout = 30    # 부드러운 종료 대기 시간
keepalive = 5            # Keep-Alive 연결 유지 시간 (초)


# ── 로깅 설정 ────────────────────────────────────────────
import logging
errorlog = "./logs/gunicorn_error.log"
accesslog = "./logs/gunicorn_access.log"
loglevel = os.getenv("LOG_LEVEL", "info").lower()

# 액세스 로그 포맷: JSON 구조화 로그
access_log_format = ('{"time":"%(t)s","remote_ip":"%(h)s",'
                     '"method":"%(m)s","url":"%(U)s",'
                     '"status":%(s)s,"bytes":%(b)s,'
                     '"duration_ms":%(D)s,"referer":"%(f)s"}')


# ── 프로세스 관리 ────────────────────────────────────────
pidfile = "./gunicorn.pid"
daemon = False     # True이면 백그라운드 데몬으로 실행


# ── 보안 설정 ────────────────────────────────────────────
forwarded_allow_ips = "*"     # Nginx 리버스 프록시 사용 시
proxy_protocol = False


# ── 워커 프로세스 초기화 훅 ──────────────────────────────
def on_starting(server):
    import os
    os.makedirs("./logs", exist_ok=True)
    print(f"Gunicorn 시작: {workers} workers, timeout={timeout}s")


def worker_init(arbiter, worker):
    """각 워커 프로세스 시작 시 실행 (GPU 초기화 등)"""
    print(f"워커 {worker.pid} 시작")


def on_exit(server):
    print("Gunicorn 종료 완료")


# ── 실행 명령어 (설정 파일 사용) ─────────────────────────
# gunicorn main:app -c gunicorn.conf.py
```

## 8.2 Hugging Face Spaces vs 자체 서버

### (1) Hugging Face Spaces의 장점과 한계

Hugging Face Spaces는 AI 모델 데모와 애플리케이션을 무료 또는 저비용으로 배포할 수 있는 호스팅 플랫폼입니다. Gradio 또는 Streamlit 앱을 저장소에 올리기만 하면 자동으로 빌드되고 공개 URL이 생성됩니다. 연구자, 학생, AI 애플리케이션 개발자들이 모델을 시연하고 공유하는 데 최적화되어 있습니다.

| 구분 | 무료 플랜 (Free) | Pro 플랜 ($9/월) | Enterprise |
| --- | --- | --- | --- |
| CPU | 2 vCPU, 16GB RAM | 8 vCPU, 32GB RAM | 커스텀 |
| GPU | ZeroGPU (큐 대기) | A10G (24GB VRAM) | A100/H100 |
| 저장공간 | 50GB | 100GB | 무제한 |
| 동시 요청 | 제한적 (큐잉) | 안정적 | SLA 보장 |
| 개인 Space | 공개만 가능 | 비공개 가능 | 비공개 |
| 커스텀 도메인 | 미지원 | 지원 | 지원 |
| Sleep 정책 | 48h 미사용 시 슬립 | 없음 | 없음 |

| 비교 항목 | Hugging Face Spaces | 자체 서버 (AWS/GCP 등) |
| --- | --- | --- |
| 초기 설정 난이도 | 매우 쉬움 (git push 한 줄) | 높음 (서버 설정, Docker, Nginx) |
| 비용 | 무료 ~ 저비용 | 중간 ~ 높음 (GPU 인스턴스 비쌈) |
| 커스터마이징 | 제한적 (Gradio/Streamlit 위주) | 완전 자유 |
| 확장성 | 제한적 (수동 플랜 업그레이드) | Auto Scaling 가능 |
| 지속 실행 | 미사용 시 슬립 (무료) | 24/7 실행 가능 |
| 보안/프라이버시 | 코드 공개 (무료) | 완전 비공개 가능 |
| 모델 접근성 | HF Hub 모델 바로 사용 | 모델 직접 다운로드/관리 |
| 권장 상황 | 학습용, 데모, 포트폴리오 | 실서비스, 기업 환경 |

```python
# app.py — Gradio 기반 이미지 설명 AI 앱 (Space용)
import gradio as gr
import torch
from transformers import AutoModelForCausalLM, AutoTokenizer
from PIL import Image


# ── 모델 로드 (Space 빌드 시 자동 실행) ─────────────────
model_id = "vikhyatk/moondream2"

tokenizer = AutoTokenizer.from_pretrained(model_id, revision="2024-08-26")
model = AutoModelForCausalLM.from_pretrained(
    model_id,
    trust_remote_code=True,
    revision="2024-08-26",
    torch_dtype=torch.float16 if torch.cuda.is_available() else torch.float32
)
model.eval()


# ── 추론 함수 ─────────────────────────────────────────────
def describe_image(image: Image.Image, question: str) -> str:
    if image is None:
        return "이미지를 업로드해주세요."
    if not question.strip():
        question = "이 이미지에 대해 설명해주세요."
    enc = model.encode_image(image)
    return model.answer_question(enc, question, tokenizer)


# ── Gradio UI ─────────────────────────────────────────────
with gr.Blocks(title="AI 이미지 설명기") as demo:
    gr.Markdown("## 🤖 AI 이미지 설명기")
    gr.Markdown("이미지를 업로드하고 질문을 입력하면 AI가 답변합니다.")
    with gr.Row():
        with gr.Column():
            img_input = gr.Image(type="pil", label="이미지 업로드")
```

## 8.3 Docker로 API 감싸기

### (1) Docker 기본 개념과 AI 서비스에서의 필요성

Docker는 애플리케이션과 그 실행 환경을 하나의 컨테이너(Container)로 패키징하는 플랫폼입니다. "내 노트북에서는 되는데 서버에서는 안 돼요" 문제를 완전히 해결합니다. AI 서비스에서 Docker가 특히 중요한 이유는 Python 패키지 버전, CUDA 버전, 시스템 라이브러리 등 의존성이 복잡하기 때문입니다.

| Docker 핵심 개념 | 설명 | 비유 |
| --- | --- | --- |
| Image (이미지) | 컨테이너의 설계도/스냅샷. 읽기 전용 레이어 구조 | 요리 레시피 |
| Container (컨테이너) | 이미지를 실행한 인스턴스. 격리된 실행 환경 | 레시피로 만든 요리 |
| Dockerfile | 이미지를 빌드하는 스크립트 파일 | 레시피 문서 |
| Registry (레지스트리) | 이미지를 저장/공유하는 저장소. Docker Hub, ECR 등 | 레시피 공유 사이트 |
| Docker Compose | 여러 컨테이너를 정의하고 함께 실행하는 도구 | 풀코스 요리 메뉴 |
| Volume (볼륨) | 컨테이너 외부에 데이터를 영구 저장하는 방법 | 냉장고 (데이터 보관) |
| Network (네트워크) | 컨테이너 간 통신을 위한 가상 네트워크 | 주방 내 통신망 |

### (2) Dockerfile: FastAPI + 모델 환경 이미지 생성

AI 서비스를 위한 Dockerfile 작성에는 여러 주요 고려사항이 있습니다. CUDA 버전 선택, 패키지 레이어 캐싱 최적화, 불필요한 파일 제외, 보안 설정 등을 체계적으로 작성해야 합니다.

```dockerfile
# Dockerfile — FastAPI AI 서비스 이미지

# ── 베이스 이미지 선택 ───────────────────────────────────
# CUDA 필요 없는 경우 (CPU 전용)
# FROM python:3.11-slim

# CUDA 12.1 + cuDNN 8 (GPU 서버 배포 시)
FROM nvidia/cuda:12.1.0-cudnn8-runtime-ubuntu22.04

# ── 환경 변수 설정 ───────────────────────────────────────
ENV PYTHONDONTWRITEBYTECODE=1 \
    PYTHONUNBUFFERED=1 \
    PYTHONFAULTHANDLER=1 \
    PIP_NO_CACHE_DIR=off \
    PIP_DISABLE_PIP_VERSION_CHECK=on \
    DEBIAN_FRONTEND=noninteractive

# ── 시스템 패키지 설치 ───────────────────────────────────
RUN apt-get update && apt-get install -y --no-install-recommends \
    python3.11 python3.11-dev python3-pip \
    build-essential curl git wget \
    libglib2.0-0 libsm6 libxext6 libxrender-dev \
    libgomp1 \
    && rm -rf /var/lib/apt/lists/*

RUN ln -sf /usr/bin/python3.11 /usr/bin/python3 \
    && ln -sf /usr/bin/python3 /usr/bin/python

# ── 작업 디렉토리 생성 ───────────────────────────────────
WORKDIR /app

# ── 보안: 비루트 사용자 생성 ─────────────────────────────
RUN groupadd --gid 1001 appuser \
    && useradd --uid 1001 --gid 1001 --create-home appuser

# ── Python 의존성 설치 ───────────────────────────────────
# requirements.txt를 먼저 복사하여 레이어 캐시 활용
COPY requirements.txt .

RUN pip install --upgrade pip \
    && pip install torch torchvision --index-url https://download.pytorch.org/whl/cu121 \
    && pip install -r requirements.txt \
    && pip install gunicorn

# ── 애플리케이션 코드 복사 ───────────────────────────────
# .dockerignore로 불필요한 파일 제외 후 복사
COPY --chown=appuser:appuser . .

# ── 로그/캐시 디렉토리 생성 ──────────────────────────────
RUN mkdir -p /app/logs /app/outputs /app/model_cache \
    && chown -R appuser:appuser /app

# ── 비루트 사용자로 전환 ─────────────────────────────────
USER appuser

# ── 포트 노출 ────────────────────────────────────────────
EXPOSE 8000

# ── 헬스체크 설정 ────────────────────────────────────────
HEALTHCHECK --interval=30s --timeout=10s --start-period=60s --retries=3 \
    CMD curl -f http://localhost:8000/health || exit 1

# ── 실행 명령 ─────────────────────────────────────────────
# 개발: uvicorn (간단하게)
# CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]

# 운영: gunicorn + uvicorn worker
CMD ["gunicorn", "main:app", \
     "--worker-class", "uvicorn.workers.UvicornWorker", \
     "--workers", "2", \
     "--bind", "0.0.0.0:8000", \
     "--timeout", "120", \
     "--access-logfile", "/app/logs/access.log", \
     "--error-logfile", "/app/logs/error.log"]
```

### (3) 모델 파일 포함 여부에 따른 이미지 최적화

AI 서비스 Docker 이미지의 가장 큰 문제는 모델 파일의 크기입니다. Stable Diffusion은 4~10GB, LLaVA-7B는 14GB를 초과합니다. 이 모델 파일을 Docker 이미지에 포함할지 외부에서 로드할지는 배포 전략에 큰 영향을 미칩니다.

| 전략 | 모델 포함 여부 | 이미지 크기 | 컨테이너 시작 시간 | 권장 상황 |
| --- | --- | --- | --- | --- |
| 전략 A: 이미지 내 포함 | O (이미지에 번들) | 매우 큼 (10~20GB) | 즉시 (다운로드 불필요) | 오프라인 환경, 엣지 배포 |
| 전략 B: HF Hub 다운로드 | X (런타임 다운로드) | 작음 (~2GB) | 느림 (첫 기동 시 수 GB 다운) | 클라우드, HF Hub 접근 가능 |
| 전략 C: Docker 볼륨 마운트 | X (외부 볼륨) | 작음 (~2GB) | 빠름 (로컬 파일) | 프로덕션 서버, 모델 공유 |
| 전략 D: S3/스토리지 다운로드 | X (S3에서 다운) | 작음 (~2GB) | 빠름 (내부 네트워크) | AWS 배포, 프라이빗 모델 |

```bash

  -e TRANSFORMERS_CACHE=/app/model_cache \
```

### (4) Docker Compose와 Gunicorn 연동

Docker Compose는 여러 컨테이너를 하나의 YAML 파일(`docker-compose.yml`)로 정의하고 함께 실행하는 도구입니다. FastAPI 앱 + Nginx 리버스 프록시 + 모니터링 도구를 한 번에 구성하는 데 유용합니다.

```yaml
# docker-compose.yml — FastAPI + Nginx + Prometheus 통합 구성
version: "3.9"

services:
  # ── FastAPI AI 앱 ────────────────────────────────────
  api:
    build:
      context: .
      dockerfile: Dockerfile
    image: ai-fastapi:latest
    container_name: ai_api
    restart: unless-stopped
    environment:
      - ENVIRONMENT=production
      - HF_TOKEN=${HF_TOKEN}
      - MODEL_CACHE_DIR=/app/model_cache
      - WORKERS=2
    volumes:
      - model_cache:/app/model_cache
      - ./logs:/app/logs
    expose:
      - "8000"           # 외부 직접 노출 안 함 (Nginx 경유)
    deploy:
      resources:
        reservations:
          devices:
            - driver: nvidia
              count: 1
              capabilities: [gpu]
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 60s
    networks:
      - ai_network

  # ── Nginx 리버스 프록시 ──────────────────────────────
  nginx:
    image: nginx:1.25-alpine
    container_name: nginx
    restart: unless-stopped
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./nginx/nginx.conf:/etc/nginx/nginx.conf:ro
      - ./nginx/ssl:/etc/nginx/ssl:ro
      - ./logs/nginx:/var/log/nginx
    depends_on:
      api:
        condition: service_healthy
    networks:
      - ai_network

  # ── Prometheus 메트릭 수집 ───────────────────────────
  prometheus:
    image: prom/prometheus:latest
    container_name: prometheus
    restart: unless-stopped
    ports:
      - "9090:9090"
    volumes:
      - ./monitoring/prometheus.yml:/etc/prometheus/prometheus.yml:ro
      - prometheus_data:/prometheus
    command:
      - "--config.file=/etc/prometheus/prometheus.yml"
      - "--storage.tsdb.retention.time=30d"
    networks:
      - ai_network

  # ── Grafana 시각화 대시보드 ──────────────────────────
  grafana:
    image: grafana/grafana:latest
    container_name: grafana
    restart: unless-stopped
    ports:
      - "3000:3000"
    environment:
      - GF_SECURITY_ADMIN_PASSWORD=${GRAFANA_PASSWORD:-admin}
    volumes:
      - grafana_data:/var/lib/grafana
      - ./monitoring/grafana/dashboards:/etc/grafana/provisioning/dashboards
    networks:
      - ai_network

volumes:
  model_cache:
    driver: local
  prometheus_data:
  grafana_data:

networks:
  ai_network:
    driver: bridge
```


### (2) SSH, 포트, 방화벽, Nginx 연동 실습

운영 환경에서 FastAPI 서비스는 직접 외부에 노출하지 않고 Nginx를 리버스 프록시로 앞에 두는 것이 표준 아키텍처입니다. Nginx가 HTTPS 종료(TLS Termination), 요청 제한(Rate Limiting), 정적 파일 서빙을 담당하고, FastAPI는 API 로직에만 집중합니다.

```nginx
# /etc/nginx/nginx.conf — Nginx 리버스 프록시 설정
user nginx;
worker_processes auto;

events {
    worker_connections 1024;
}

http {
    include      /etc/nginx/mime.types;
    default_type application/octet-stream;

    # ── 보안 헤더 ─────────────────────────────────────
    add_header X-Frame-Options        "SAMEORIGIN"    always;
    add_header X-Content-Type-Options "nosniff"       always;
    add_header X-XSS-Protection        "1; mode=block" always;
    add_header Referrer-Policy         "strict-origin" always;

    # ── 요청 크기 제한 ────────────────────────────────
    client_max_body_size 25M;    # 이미지 업로드 허용
    client_body_timeout 120s;

    # ── 압축 설정 ─────────────────────────────────────
    gzip on;
    gzip_types application/json text/plain;

    # ── Rate Limiting ─────────────────────────────────
    limit_req_zone $binary_remote_addr zone=api_limit:10m rate=10r/s;

    upstream fastapi_backend {
        server api:8000;
        keepalive 32;
    }

    # ── HTTP → HTTPS 리다이렉트 ──────────────────────
    server {
        listen 80;
        server_name myapp.com www.myapp.com;
        return 301 https://$server_name$request_uri;
    }

    # ── HTTPS 서버 ────────────────────────────────────
    server {
        listen 443 ssl http2;
        server_name myapp.com www.myapp.com;

        ssl_certificate     /etc/nginx/ssl/fullchain.pem;
        ssl_certificate_key /etc/nginx/ssl/privkey.pem;
        ssl_protocols       TLSv1.2 TLSv1.3;
        ssl_ciphers         HIGH:!aNULL:!MD5;

        # ── API 프록시 ────────────────────────────────
        location /api/ {
            limit_req zone=api_limit burst=20 nodelay;
            proxy_pass          http://fastapi_backend;
            proxy_set_header    Host              $host;
            proxy_set_header    X-Real-IP         $remote_addr;
            proxy_set_header    X-Forwarded-For   $proxy_add_x_forwarded_for;
            proxy_set_header    X-Forwarded-Proto $scheme;
            proxy_read_timeout  120s;
            proxy_send_timeout  120s;
            proxy_request_buffering off;
        }

        # ── Swagger UI 접근 ───────────────────────────
        location /docs {
            proxy_pass http://fastapi_backend;
            proxy_set_header Host $host;
        }

        # ── 헬스체크 ──────────────────────────────────
        location /health {
            proxy_pass http://fastapi_backend;
            access_log off;
        }
    }
}
```

## 8.5 API 모니터링과 속도 최적화

### (1) 요청/응답 로그 남기기 (LoggingMiddleware)

운영 중인 API에서 발생하는 요청과 응답을 체계적으로 로깅하는 것은 장애 추적, 성능 분석, 보안 감사에 필수적입니다. FastAPI의 미들웨어(Middleware) 기능을 활용하면 모든 요청에 대해 일관된 로그를 남길 수 있습니다.

```python
# logging_middleware.py — 요청/응답 로깅 미들웨어
import time, uuid, json
from fastapi import Request, Response
from starlette.middleware.base import BaseHTTPMiddleware
from loguru import logger


class RequestLoggingMiddleware(BaseHTTPMiddleware):
    """모든 HTTP 요청/응답을 구조화된 JSON으로 로깅합니다."""

    # 로깅 제외 경로 (헬스체크, 메트릭 등)
    SKIP_PATHS = {"/health", "/metrics", "/favicon.ico"}

    async def dispatch(self, request: Request, call_next) -> Response:
        # 헬스체크 경로 스킵
        if request.url.path in self.SKIP_PATHS:
            return await call_next(request)

        # 요청 ID 생성 (추적용)
        request_id = str(uuid.uuid4())[:8]
        request.state.request_id = request_id

        # 요청 정보 수집
        start_time = time.time()
        client_ip = request.headers.get("X-Real-IP") or \
            request.headers.get("X-Forwarded-For", "").split(",")[0].strip() or \
            (request.client.host if request.client else "unknown")

        log_data = {
            "request_id": request_id,
            "method":     request.method,
            "path":       request.url.path,
            "query":      str(request.url.query) or None,
            "client_ip":  client_ip,
            "user_agent": request.headers.get("User-Agent", "")[:100],
        }

        # 요청 처리
        try:
            response = await call_next(request)
            status_code = response.status_code
        except Exception as exc:
            elapsed_ms = (time.time() - start_time) * 1000
            logger.error({
                **log_data,
                "status": 500,
                "elapsed_ms": round(elapsed_ms, 2),
                "error": str(exc)
            })
            raise

        # 응답 로깅
        elapsed_ms = (time.time() - start_time) * 1000
        log_entry = {
            **log_data,
            "status": status_code,
            "elapsed_ms": round(elapsed_ms, 2),
        }

        # 상태 코드에 따라 로그 레벨 분류
        if status_code >= 500:
            logger.error(json.dumps(log_entry, ensure_ascii=False))
        elif status_code >= 400:
            logger.warning(json.dumps(log_entry, ensure_ascii=False))
        elif elapsed_ms > 5000:
            logger.warning(json.dumps({**log_entry, "slow_request": True}, ensure_ascii=False))
        else:
            logger.info(json.dumps(log_entry, ensure_ascii=False))

        # 응답 헤더에 요청 ID 추가
        response.headers["X-Request-ID"] = request_id
        response.headers["X-Response-Time"] = f"{elapsed_ms:.2f}ms"
        return response


# ── loguru 설정 (main.py에서 호출) ──────────────────────
from loguru import logger
import sys


def setup_logging(log_file: str = "./logs/app.log"):
    """loguru 핸들러 설정"""
    import os
    os.makedirs(os.path.dirname(log_file), exist_ok=True)

    # 기본 핸들러 제거
    logger.remove()

    # 콘솔 출력 (색상 포함)
    logger.add(sys.stderr, level="DEBUG" if os.getenv("DEBUG") else "INFO",
               format="<green>{time:HH:mm:ss}</green> | <level>{level:7}</level> | {message}")

    # 파일 로테이션 (하루 단위, 최대 30일 보관)
    logger.add(log_file,
               rotation="00:00",      # 자정마다 새 파일
               retention="30 days",   # 30일 후 자동 삭제
               compression="gz",      # 압축 보관
               level="INFO",
               encoding="utf-8",
               serialize=True)        # JSON 형식으로 저장
```

Uvicorn과 Gunicorn의 로그 설정을 통해 서버 레벨의 액세스 로그와 에러 로그를 체계적으로 관리합니다. JSON 형식의 구조화 로그는 Elasticsearch, Loki 등 로그 수집 시스템과 쉽게 통합됩니다.

```python
import json
from datetime import datetime


class UvicornJSONFormatter(logging.Formatter):
    """Uvicorn 로그를 JSON 형식으로 출력하는 포매터"""

    def format(self, record: logging.LogRecord) -> str:
        log_data = {
            "timestamp": datetime.utcnow().isoformat() + "Z",
            "level":     record.levelname,
            "logger":    record.name,
            "message":   record.getMessage(),
        }
        if record.exc_info:
            log_data["exception"] = self.formatException(record.exc_info)
        return json.dumps(log_data, ensure_ascii=False)
```

### (3)(4) 응답 속도 측정 및 개선 전략

AI API의 응답 속도는 사용자 경험에 직결됩니다. 느린 응답의 원인을 정확히 파악하고 단계별로 최적화하는 체계적인 접근이 필요합니다.

```python
# performance_profiler.py — 엔드포인트 성능 측정 도구
import time, asyncio
import functools
from fastapi import Request
from loguru import logger


# ── 데코레이터 기반 함수 수준 프로파일링 ─────────────────
def profile_async(func):
    """비동기 함수의 실행 시간을 측정하는 데코레이터"""
    @functools.wraps(func)
    async def wrapper(*args, **kwargs):
        start = time.perf_counter()
        result = await func(*args, **kwargs)
        elapsed = (time.perf_counter() - start) * 1000
        logger.debug(f"[PROFILE] {func.__name__}: {elapsed:.2f}ms")
        if elapsed > 1000:
            logger.warning(f"[SLOW] {func.__name__}: {elapsed:.2f}ms 초과")
        return result
    return wrapper


# ── 단계별 시간 측정 컨텍스트 매니저 ─────────────────────
class Timer:
    """코드 블록의 실행 시간을 측정합니다."""

    def __init__(self, label: str = ""):
        self.label = label
        self.checkpoints = {}
        self._start = None

    def start(self):
        self._start = time.perf_counter()
        return self

    def checkpoint(self, name: str):
        """중간 지점 시간 기록"""
        self.checkpoints[name] = (time.perf_counter() - self._start) * 1000

    def stop(self) -> dict:
        total = (time.perf_counter() - self._start) * 1000
        return {
            "label": self.label,
            "total_ms": round(total, 2),
            "checkpoints": {k: round(v, 2) for k, v in self.checkpoints.items()}
        }


# ── 실제 사용 예시 ───────────────────────────────────────
@app.post("/api/v1/generate")
async def generate_image(request: ImageGenRequest):
    timer = Timer("image_generation").start()

    # 단계 1: 입력 검증
    validated = validate_request(request)
    timer.checkpoint("validation")

    # 단계 2: 전처리
    processed = preprocess(validated)
    timer.checkpoint("preprocessing")

    # 단계 3: 모델 추론 (가장 오래 걸리는 단계)
    result = await run_inference(processed)
    timer.checkpoint("inference")

    # 단계 4: 후처리
    response = postprocess(result)
    timer.checkpoint("postprocessing")

    timing = timer.stop()
    logger.info(f"타이밍: {timing}")
    # 출력 예: {"label":"image_generation","total_ms":3420.15,
    # "checkpoints":{"validation":2.1,"preprocessing":45.3,
    # "inference":3350.2,"postprocessing":22.5}}

    return response
```

| --- | --- | --- | --- |
| 모델 로딩 지연 | 첫 요청만 30~60초 느림 | 모델 지연 로드(lazy load) | lifespan에서 미리 로드 + 워밍업 추론 |
| GPU 메모리 부족 | 간헐적 OOM 에러 | 동시 요청이 VRAM 초과 | inference_lock으로 직렬화, 워커 수 감소 |

| 디스크 I/O | 이미지 저장/읽기 느림 | 동기적 파일 I/O | aiofiles로 비동기 처리, 메모리 처리 |

```python
# pip install prometheus-fastapi-instrumentator

from fastapi import FastAPI
from prometheus_fastapi_instrumentator import Instrumentator, metrics
from prometheus_client import Counter, Histogram, Gauge
import torch


# ── 커스텀 메트릭 정의 ───────────────────────────────────
MODEL_INFERENCE_TIME = Histogram(
    "ai_model_inference_seconds",
    "AI 모델 추론 시간 (초)",
    buckets=[0.1, 0.5, 1.0, 2.0, 5.0, 10.0, 30.0, 60.0]
)

MODEL_REQUESTS_TOTAL = Counter(
    "ai_model_requests_total",
    "AI 모델 총 요청 수",
    labelnames=["model_name", "status"]
)

GPU_MEMORY_USED = Gauge(
    "gpu_memory_used_bytes",
    "GPU 메모리 사용량 (bytes)",
    labelnames=["device"]
)

ACTIVE_REQUESTS = Gauge(
    "ai_active_requests",
    "현재 처리 중인 요청 수"
)


def setup_prometheus(app: FastAPI) -> None:
    """FastAPI 앱에 Prometheus 계측기를 부착합니다."""

    instrumentator = Instrumentator(
        should_group_status_codes=False,
        should_respect_env_var=True,
        env_var_name="ENABLE_METRICS",
        excluded_handlers=["/health", "/metrics"],
    )

    # 기본 HTTP 메트릭 추가
    instrumentator.add(metrics.request_size())
    instrumentator.add(metrics.response_size())
    instrumentator.add(metrics.latency(buckets=[0.1, 0.5, 1, 5, 10, 30]))

    # /metrics 엔드포인트 등록 및 계측 시작
    instrumentator.instrument(app).expose(app, endpoint="/metrics")


# ── GPU 메트릭 주기적 업데이트 ──────────────────────────
async def update_gpu_metrics():
    """GPU 메모리 사용량을 주기적으로 Prometheus에 기록합니다."""
    while True:
        if torch.cuda.is_available():
            for i in range(torch.cuda.device_count()):
                used = torch.cuda.memory_allocated(i)
                GPU_MEMORY_USED.labels(device=f"cuda:{i}").set(used)
        await asyncio.sleep(15)     # 15초마다 갱신
```

```python
        ).inc()
        return result
    except Exception as e:
        MODEL_REQUESTS_TOTAL.labels(
            model_name="stable-diffusion", status="error"
        ).inc()
        # Sentry로 에러 자동 전송
        raise
    finally:
        elapsed = time.time() - start
        MODEL_INFERENCE_TIME.observe(elapsed)
        ACTIVE_REQUESTS.dec()
```
