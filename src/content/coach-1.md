# 1회차 · AI 서비스 아키텍처 코칭

프로젝트의 전체 아키텍처를 검토하고 개선 방향을 코칭합니다.

## 코칭 목표

- 프로젝트 아키텍처 리뷰
- 기술 스택 적합성 평가
- LLM API 활용 전략 수립

## 주요 내용

각 팀의 프로젝트 구조를 전문가와 함께 검토합니다. 프론트엔드-백엔드-AI 연동 아키텍처의 적절성을 평가하고, 개선점을 제안합니다.

## 체크포인트

- 컴포넌트 구조가 적절한가?
- API 호출 패턴이 효율적인가?
- 에러 처리가 충분한가?
- 보안 취약점은 없는가?

## 좋은 아키텍처의 5가지 기준

- 관심사 분리 — 화면(React)·데이터(Supabase)·AI(Edge Function)가 각자 일한다
- 비밀 분리 — API 키는 프론트에 없고 Edge Function/Secrets에만
- 상태 일관성 — 로딩·성공·에러를 같은 방식으로 다룬다
- 에러·로딩 표준 — 모든 비동기 호출에 둘 다 처리
- 확장 여지 — 기능 추가가 기존 코드를 깨지 않는 구조

## 레이어 점검표

| 레이어 | 책임 | 흔한 실수 |
| --- | --- | --- |
| 프론트(React) | 화면·입력·상태 | 컴포넌트 비대·중복 API 호출 |
| 데이터(Supabase) | 저장·인증·RLS | RLS 누락으로 전체 노출 |
| AI(Edge Function) | LLM 호출·키 보호 | 프론트에서 키 들고 직접 호출 |

## 리팩터링 신호 (이러면 구조를 손볼 때)

- 같은 코드를 세 번째 복붙하고 있다
- 한 컴포넌트가 300줄을 넘는다
- props를 4~5단계 아래로 전달한다
- 한 함수가 화면·통신·검증을 다 한다
